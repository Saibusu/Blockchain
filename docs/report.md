## 一、前言

### 1.1 研究背景與動機

近年來，全球食品安全事件層出不窮，從台灣的黑心油品事件、歐洲的馬肉摻偽醜聞，到中國的毒奶粉事件，無不顯示現行食品供應鏈在透明度與可信度上的嚴重缺陷。消費者越來越難以確認食品的真實產地、生產過程與運輸條件，而企業與監管機關亦難以在食安事件爆發後迅速追查問題根源。

與此同時，區塊鏈技術自比特幣問世以來，逐漸從加密貨幣領域擴展至各產業應用，其去中心化、不可竄改、可追溯等特性，使其成為解決食品供應鏈痛點的潛力技術。本研究旨在探討區塊鏈技術如何有效改善食品供應鏈的透明度、安全性與效率，並透過概念驗證系統（PoC）展示其實際可行性。

### 1.2 研究目的與範圍

本研究的主要目的如下：

1. 梳理現行食品供應鏈的運作模式與痛點問題
2. 探討區塊鏈技術導入食品供應鏈的可行方案與潛在價值
3. 分析此應用場景面臨的機遇與挑戰
4. 實作一套基於以太坊智能合約的食品溯源概念驗證系統

研究範圍聚焦於從農產品生產端至終端消費者的完整供應鏈流程，以台灣本地農產品（如蔬果、肉品）為主要情境案例。

## 二、食品供應鏈業務情境

### 2.1 食品供應鏈運作流程概述

食品供應鏈是一個涵蓋多個環節、多方參與者的複雜系統，其流程可分為以下五個主要階段：

| 階段 | 參與者 | 主要任務 |
|------|--------|----------|
| 生產端 | 農民 / 農場 | 種植養殖、記錄農藥施肥、採收 |
| 加工包裝端 | 食品加工廠 | 清洗分級、加工、包裝 |
| 物流倉儲端 | 物流商 | 冷鏈運輸、溫濕度監控、配送 |
| 零售端 | 超市 / 市場 | 陳列販售、保鮮期管理 |
| 消費端 | 消費者 | 購買、查詢食品履歷 |

**流程示意：**

```
[農場/產地] → [加工廠] → [物流倉儲] → [零售商] → [消費者]
     ↑              ↑            ↑            ↑           ↑
  登記批次      記錄加工      記錄溫度      確認收貨     掃碼查詢
```

### 2.2 主要參與者與角色

| 參與者 | 主要職責 | 資料需求 |
|--------|----------|----------|
| 農民 / 生產者 | 生產、記錄產地資訊 | 寫入產品基本資料 |
| 加工廠 | 加工、包裝、品管 | 記錄加工批次與成分 |
| 物流商 | 運輸、倉儲管理 | 記錄溫度、位置、交接時間 |
| 零售商 | 陳列、銷售 | 讀取商品履歷、更新庫存狀態 |
| 消費者 | 購買、查詢資訊 | 查詢完整供應鏈履歷 |
| 監管機關 | 稽查、認證 | 稽核所有環節資料 |

### 2.3 典型業務場景

以台灣有機蔬菜供應鏈為例：宜蘭有機農場種植高麗菜，採收後由物流業者以冷藏車運送至台北加工廠進行清洗包裝，再配送至各大超市。消費者在超市購買時，透過包裝上的 QR Code 掃描即可查詢從農場到貨架的完整歷程。

## 三、食品供應鏈現狀與關鍵問題

### 3.1 資訊不透明與資料孤島問題

現行食品供應鏈中，各參與者各自維護獨立的資料系統，農民使用農務紀錄本、加工廠使用 ERP 系統、物流商使用 GPS 追蹤系統，這些系統彼此不互通，形成「**資料孤島**」（Data Silos）現象。當食安問題發生時，跨系統的資料整合曠日費時，嚴重延誤處理時機。

### 3.2 假冒偽劣與產地造假

產地標示造假是食品業長期存在的問題。不肖業者可能以廉價進口食材混充本地有機農產品，或偽造有機認證標章。由於現行紙本憑證容易偽造，消費者與監管機關難以有效辨別真偽。根據調查，**全球食品詐欺每年造成超過 400 億美元的經濟損失**。

### 3.3 食安事件應對遲緩

當食品安全事件爆發時，追查問題源頭往往需要數天甚至數週。以 2011 年台灣塑化劑事件為例，事件爆發後相關機關花費大量人力物力才逐步釐清受污染的產品範圍，期間已造成難以估計的公共健康風險。

### 3.4 冷鏈管理難以稽核

冷凍食品、生鮮肉品等需要全程冷鏈運輸的商品，其溫度記錄目前多依賴物流商自行填報，缺乏客觀可信的第三方驗證機制。一旦冷鏈斷鏈，責任歸屬往往成為各方推諉的爭議焦點。

### 3.5 人工作業效率低落

供應鏈各環節大量依賴紙本文件與人工核對，不僅耗費人力，更容易因人為疏失造成資料錯誤。進出口食品還需應付各國不同的法規文件要求，流程繁複且容易出錯。

## 四、區塊鏈解決方案的價值與優勢

### 4.1 區塊鏈技術核心特性

| 特性 | 說明 | 對食品供應鏈的意義 |
|------|------|------------------|
| 去中心化 | 資料分散於多個節點 | 無單一控制方，各方地位對等 |
| 不可竄改 | 密碼學雜湊鎖定資料 | 杜絕產地造假、記錄竄改 |
| 透明可追溯 | 全程履歷公開可查 | 消費者可即時查詢食品來源 |
| 智能合約 | 自動化執行業務邏輯 | 自動付款、超溫警報、認證管理 |

### 4.2 食品溯源全程上鏈

透過區塊鏈技術，食品供應鏈的每一個環節均可將關鍵資訊寫入鏈上，形成完整且不可竄改的食品履歷。消費者只需掃描 QR Code，即可在數秒內查閱從農場到貨架的完整歷程。

> **實際案例：** 沃爾瑪（Walmart）與 IBM 合作導入基於 Hyperledger Fabric 的食品溯源系統後，芒果的產地追溯時間從原本的 **6.5 天縮短至 2.2 秒**。

### 4.3 智能合約實現自動化流程

智能合約可自動化供應鏈中的多項業務流程：

- **自動付款**：物流商完成交貨並確認後，智能合約自動觸發付款，減少帳款糾紛
- **品質驗核**：IoT 感測器偵測到冷鏈溫度超標時，智能合約自動記錄異常並通知相關方
- **認證管理**：有機認證文件的有效期限由智能合約自動管理，到期自動失效並提醒更新

### 4.4 建立多方信任機制

區塊鏈的去中心化信任機制使各方無需完全信任彼此，只需信任共同遵守的程式規則（智能合約），大幅降低信任成本，解決多方協作的信任障礙。

### 4.5 提升消費者透明度

消費者可透過手機 App 或網頁，以商品條碼或 QR Code 查詢：

- 產地農場資訊（位置、認證）
- 農藥使用紀錄
- 加工廠資訊與衛生稽核結果
- 物流溫度記錄
- 碳足跡資訊

## 五、機遇與挑戰

### 5.1 機遇分析

| 機遇 | 說明 |
|------|------|
| 政策法規推動 | 台灣食安法對追溯追蹤制度有明確規範，提供政策需求基礎 |
| 消費者意識提升 | 超過七成消費者表示願意為可溯源食品支付更高價格 |
| IoT 技術成熟 | RFID、GPS、溫度感測器日益普及，解決鏈上鏈下串接問題 |
| 成功案例佐證 | 沃爾瑪、阿里巴巴、雀巢等企業已有可供參考的成功案例 |

### 5.2 挑戰分析

#### （1）鏈上鏈下資料一致性（Oracle Problem）

區塊鏈能確保鏈上資料的不可竄改性，但若源頭輸入的資料本身即為假造，區塊鏈無法自動辨別。需結合 IoT 感測器、防偽標籤等手段加以輔助。

#### （2）導入成本與中小企業門檻

食品供應鏈中有大量中小型農場與加工廠，其資訊化程度低，導入需投入設備採購、系統整合與人員培訓等成本，可能形成採用障礙。

#### （3）各方協作意願

供應鏈系統的價值在於各方共同參與，若部分環節拒絕上鏈，整體溯源效果將大打折扣。如何形成產業聯盟、建立共同標準是推廣的關鍵挑戰。

#### （4）資料隱私與商業機密

採購價格、供應商名單等商業機密資訊若完全公開，可能引發競爭疑慮。需透過許可式區塊鏈（Permissioned Blockchain）或零知識證明（Zero-Knowledge Proof）等技術進行分層存取控制。

#### （5）效能與擴展性

公鏈（如以太坊）每秒處理的交易量有限，對大規模供應鏈可能存在壅塞與高手續費問題，需考慮採用聯盟鏈或 Layer 2 解決方案。

## 六、技術實作（Proof of Concept）

### 6.1 系統設計概述

本 PoC 系統以以太坊區塊鏈為基礎，使用 **Remix IDE** 進行開發，模擬食品供應鏈中從生產端到消費端的完整溯源流程。系統核心為一個以 **Solidity** 編寫的智能合約，實現食品批次的登記、各環節交接記錄上鏈與消費者查詢等功能。

**開發環境：**

| 項目 | 內容 |
|------|------|
| 區塊鏈平台 | 以太坊（Ethereum） |
| 開發工具 | Remix IDE |
| 程式語言 | Solidity ^0.8.0 |
| 測試環境 | JavaScript VM（Remix 內建） |
| 合約檔案 | `contracts/FoodTraceability.sol` |

### 6.2 合約架構

```
FoodTraceability Contract
├── 資料結構（Structs）
│   ├── FoodBatch        — 食品批次基本資料
│   └── TraceRecord      — 單筆供應鏈追蹤記錄
├── 角色管理（Role Management）
│   ├── Producer         — 生產者
│   ├── Processor        — 加工商
│   ├── Logistics        — 物流商
│   ├── Retailer         — 零售商
│   └── Regulator        — 監管機關
├── 核心函式（Functions）
│   ├── registerBatch()      ← 登記食品批次（Producer）
│   ├── addTraceRecord()     ← 新增供應鏈環節記錄
│   ├── deactivateBatch()    ← 停用批次/產品召回（Regulator）
│   ├── getBatchInfo()       ← 查詢批次基本資訊
│   ├── getBatchHistory()    ← 查詢完整批次履歷
│   └── getTraceCount()      ← 查詢記錄筆數
└── 事件（Events）
    ├── BatchRegistered      — 批次登記成功
    ├── RecordAdded          — 新增環節記錄
    ├── TemperatureAlert     — 冷鏈超溫警報
    └── RoleAssigned         — 角色指派
```

### 6.3 智能合約程式碼

> 完整程式碼請見 [`contracts/FoodTraceability.sol`](contracts/FoodTraceability.sol)

以下列出核心功能片段：

**批次登記（Producer 呼叫）：**

```solidity
function registerBatch(
    string memory _batchId,
    string memory _productName,
    string memory _origin,
    uint256 _productionDate
) external onlyRole(Role.Producer) {
    require(!batches[_batchId].isActive, "Batch ID already registered");

    batches[_batchId] = FoodBatch({
        batchId:        _batchId,
        productName:    _productName,
        origin:         _origin,
        productionDate: _productionDate,
        producer:       msg.sender,
        isActive:       true
    });
    // 自動建立第一筆產地記錄...
    emit BatchRegistered(_batchId, _productName, msg.sender);
}
```

**冷鏈超溫自動警報：**

```solidity
// 冷鏈超溫警告：超過 8.0°C（即 _temperature > 80）自動觸發
if (_temperature > 80) {
    emit TemperatureAlert(_batchId, _temperature, msg.sender);
}
```

### 6.4 操作流程說明

**步驟一：部署合約**
在 Remix IDE 中選擇編譯器版本 `0.8.x`，編譯後部署至 JavaScript VM（測試環境）。

**步驟二：角色指派**
由合約擁有者（owner）呼叫 `assignRole()`，分別指派不同帳號為生產者、加工商、物流商與零售商角色。

| 角色 | Role 數值 |
|------|-----------|
| Producer（生產者） | 1 |
| Processor（加工商） | 2 |
| Logistics（物流商） | 3 |
| Retailer（零售商） | 4 |
| Regulator（監管機關） | 5 |

**步驟三：登記食品批次**
切換至生產者帳號，呼叫 `registerBatch()`：

```
batchId:        "BATCH-2025-001"
productName:    "有機高麗菜"
origin:         "宜蘭縣員山鄉"
productionDate: 1748131200   （2025-05-25 的 Unix timestamp）
```

**步驟四：各環節上鏈**
依序切換至加工商、物流商、零售商帳號，呼叫 `addTraceRecord()` 紀錄各自的操作資訊與溫度數據。

| 操作者 | location | action | temperature |
|--------|----------|--------|-------------|
| 加工廠 | 台北市大同區加工廠 | 清洗分級包裝完成 | 40（4.0°C） |
| 物流商 | 國道一號冷藏車 | 低溫冷藏運輸中 | 42（4.2°C） |
| 零售商 | 新北市板橋全聯超市 | 商品上架完成 | 60（6.0°C） |

**步驟五：消費者查詢**
任何帳號均可呼叫 `getBatchHistory("BATCH-2025-001")` 查詢完整供應鏈記錄。

### 6.5 關鍵功能展示

| 功能 | 呼叫函式 | 說明 |
|------|----------|------|
| 批次登記 | `registerBatch()` | 生產者建立食品批次基本資料 |
| 環節上鏈 | `addTraceRecord()` | 各參與者記錄操作資訊 |
| 溫度警報 | 自動觸發 Event | 超溫時自動發出 `TemperatureAlert` |
| 產品召回 | `deactivateBatch()` | 監管機關停用問題批次 |
| 履歷查詢 | `getBatchHistory()` | 任何人均可查詢完整歷程 |
| 角色管理 | `assignRole()` | 管理員管理參與者權限 |

## 七、結論

### 7.1 研究總結

本研究透過文獻探討與技術實作，驗證了區塊鏈技術應用於食品供應鏈的可行性與潛在價值。區塊鏈的不可竄改性與透明可追溯特性，能有效解決食品供應鏈中長期存在的資訊不透明、產地造假與食安事件應對遲緩等問題；智能合約的自動化能力則能提升各環節的業務效率，降低人工成本與信任摩擦。

然而，技術本身並非萬能，鏈上鏈下的資料一致性問題、導入成本以及各方協作意願，仍是實際推廣時需要克服的核心挑戰。

### 7.2 未來展望

1. **結合 IoT 裝置**：整合 RFID 與溫度感測器，實現自動化的鏈下資料採集，降低人為輸入錯誤的風險
2. **採用聯盟鏈架構**：以 Hyperledger Fabric 取代公鏈，在保留區塊鏈核心優勢的同時，兼顧效能、隱私與商業機密保護
3. **建立產業聯盟標準**：推動食品供應鏈各方採用統一的資料格式與上鏈標準，擴大生態系規模
4. **消費者端 App 開發**：開發友善的消費者介面，讓查詢食品履歷成為日常購物的標準流程
